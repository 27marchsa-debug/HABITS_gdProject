## Ground Rule
1. 회의 

·정기 회의는 매주 수요일 17:00~ / 목요일 15:30~에 진행한다.
·회의 전 각자 진행 사항을 문서로 정리하고 충분히 숙지한다.
·스프린트 플래닝은 매주 수요일 정기 회의에서 진행하며, 역할 분담·마일스톤 검토·추가 작업 정리를 포함한다.
·스프린트 회고는 플래닝 직전에 10~15분간 진행하며, 회고 내용은 Ideation.md에 간략히 기록한다.
·회의는 개선 중심, 명확·간결한 의사소통, 시간 준수를 원칙으로 한다.

2. 절차 
·전체 개발 절차는 스크럼·애자일 방식을 기본으로 한다.
·다음 단계로의 진전은 **실현 가능성(feasibility)**을 중점으로 판단한다.
·기능 개발은 Issue → Branch → Commit → PR → Review → Merge 흐름을 따른다.
·PR은 팀원 리뷰 후 머지한다
.오류 또는 긴급 이슈 발생 시 즉시 팀 채널(카톡/노션/슬랙)을 통해 공유한다.

4. 문서화
·프로젝트 로드맵은 가능한 상세하게 작성하여 방향성을 명확히 한다.
·매주 스크럼 회의 내용을 Notion에 기록하고, 핵심을 추려 Ideation.md에 업데이트한다.
·데이터 구조, API 명세, 아키텍처 등 기술 문서는 변경 사항 발생 시 즉시 업데이트한다.
·문서 작성 원칙: 명확성, 일관성, 최신성 유지

5. 백로그 
·프로젝트 백로그는 Notion 페이지에서 통합 관리한다.
·새로운 작업 또는 요구사항 변경 시 백로그 업데이트 후 우선순위 조정 및 마일스톤 반영을 진행한다.
·각 백로그 항목은 다음을 포함한다:
          ·작업 제목
           상세 내용
           담당자
           예상 난이도
           예상 완료일
           완료된 작업은 회의에서 검증 후 Done 처리한다.

6. 협업 규칙 
·중요한 논의는 반드시 기록으로 남긴다.
·팀원 간 응답은 가능하면 24시간 이내 처리한다.
·코드 컨벤션을 준수하며 테스트 가능하고 재사용성을 고려한 코드를 작성한다.
·각자 맡은 업무는 책임감을 갖고 일정 내에 수행하며 문제 발생 시 즉시 팀과 공유한다.

7. 리스크 관리
·일정 지연 위험, 기술적 난관 등 리스크는 주간 회의에서 공유하여 사전 대응한다.
·예기치 않은 문제 발생 시 일정 또는 개발 계획을 유연하게 조정할 수 있다.
·의사 결정이 필요한 사안은 다수 의견 또는 팀 리더 판단으로 신속하게 결정한다.
